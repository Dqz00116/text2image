## Description

Non-multimodal models (text-only LLMs) cannot "see" generated images to evaluate and tune prompts. This feature adds a `describe` subcommand that uses an external vision-capable backend to describe images, enabling prompt tuning feedback loops for text-only models.

Two backends are supported via a unified adapter interface:

1. **Kimi-Code CLI** — local `kimi --print` subprocess call (non-interactive, free/本地)
2. **OpenRouter Vision Model** — API call through existing OpenAI SDK (uses API credits but no local install)

Users pick the backend via `--backend kimi` (default) or `--backend openrouter`.

## Architecture: Adapter Layer

```
cli.py (Click group)
  ├── generate  (existing, unchanged behavior)
  └── describe  (new)
        │
        ▼
ImageDescriber (Protocol / ABC)
  ├── KimiCodeDescriber     → subprocess: kimi --print
  └── OpenRouterVisionDescriber → SDK: chat.completions.create
```

Both implement: `describe(image_path: str, prompt: str) -> DescribeResult` where:

```python
class DescribeResult(TypedDict):
    description: str       # The text description/answer
    model: str             # Backend model used
    backend: str           # "kimi" | "openrouter"
    usage: dict | None     # Token usage (None for Kimi backend)
```

## CLI Design

Refactor into Click subcommand group:

```
cli.py generate  → original image generation (current behavior)
cli.py describe IMAGE [PROMPT] [options]  → image description
```

`describe` accepts:
- `IMAGE` (required, positional): path to image file
- `PROMPT` (optional, positional): custom question about the image
  - Default: "Please describe this image in detail, focusing on style, composition, colors, subjects, and mood."
- `--backend` (choice: kimi|openrouter, default: kimi): vision backend
- `--timeout` (int): timeout seconds, priority: `--timeout` > `DESCRIBE_TIMEOUT` env/`.env` > default 60
- `--json` (flag): structured JSON output for programmatic use

Backend-specific config:
- Kimi: `KIMI_PATH` env var (default: `kimi` from PATH)
- OpenRouter: reuses existing `OPENROUTER_API_KEY` env var / `.env`
- OpenRouter vision model: configurable via `--vision-model` (default: `openai/gpt-4o`)
- Timeout: `DESCRIBE_TIMEOUT` env var / `.env` (default: 60)

## User Story

```
# Generate
uv run python cli.py generate -p "a cat in watercolor" -o cat.png

# Describe with Kimi (default, local)
uv run python cli.py describe cat.png
# → "A watercolor painting of a ginger cat..."

# Describe with custom question
uv run python cli.py describe cat.png "有几个角色"

# Describe with OpenRouter vision
uv run python cli.py describe cat.png --backend openrouter

# Describe with JSON output (parsed, clean JSON)
uv run python cli.py describe cat.png --json
# → {"description": "A watercolor painting...", "model": "kimi", "backend": "kimi", "usage": null}
```

## Acceptance Criteria

- [ ] CLI refactored to Click group: `cli` group with `generate` and `describe` subcommands
- [ ] `generate` subcommand preserves all current options and behavior (backward compatible)
- [ ] `describe` subcommand: `IMAGE` (required positional), `PROMPT` (optional positional)
- [ ] `ImageDescriber` protocol class defining `describe(image_path, prompt) -> dict`
- [ ] `KimiCodeDescriber`: subprocess calls `kimi --print --final-message-only`
- [ ] `OpenRouterVisionDescriber`: uses existing OpenAI SDK to call vision model
- [ ] Backend selection via `--backend` flag
- [ ] JSON output mode (`--json`) for programmatic consumption
- [ ] Timeout handling (configurable `--timeout`, default 60s)
- [ ] Graceful error when `kimi` is not installed (for kimi backend)
- [ ] Graceful error when API key is missing (for openrouter backend)
- [ ] All existing tests pass (no regressions)
- [ ] Unit tests for both describers with mocked subprocess / API
- [ ] Unit tests for adapter selection logic
- [ ] Integration test for `describe` command

## Notes

- Kimi-Code Print mode: https://www.kimi-cli.com/zh/customization/print-mode.html
- OpenRouter vision: standard chat completions with `image_url` content blocks (already supported by existing `_build_messages`)
- `kimi --print` exit codes: 0=success, 1=fatal, 75=retryable
- Existing `TextToImageClient._build_messages()` already handles image → data URL conversion
