# Design: Visual Description Backend

## Approach A: Protocol-based Adapter Layer (推荐)

独立的 `describer.py` 模块，Protocol 定义接口，每个后端独立实现，factory 函数按 `--backend` 选择。

```
src/text2image/describer.py
├── DescribeResult (TypedDict)
├── ImageDescriber (ABC)
│   ├── describe(image_path, prompt) -> DescribeResult
├── KimiCodeDescriber(ImageDescriber)
├── OpenRouterVisionDescriber(ImageDescriber)
└── create_describer(backend) -> ImageDescriber
```

**Pros:**
- 每个后端独立可测（mock subprocess / mock API）
- 新增后端只需加一个类，不开膛现有代码
- 接口明确，IDE 类型检查友好

**Cons:**
- 3 个类 vs 1 个，文件更长

**Effort:** 2-3 hours

---

## Approach B: Single Class with Conditional Branching

单一 `ImageDescriber` 类，内部 `if/else` 分支。

```python
class ImageDescriber:
    def describe(self, image_path, prompt):
        if self.backend == "kimi":
            return self._describe_via_kimi(image_path, prompt)
        else:
            return self._describe_via_openrouter(image_path, prompt)
```

**Pros:**
- 一个类，文件短
- 初始化逻辑更紧凑

**Cons:**
- 单元测试需要同时 mock subprocess 和 API
- 加第三个后端时代码膨胀明显
- 两个后端的内部方法混在一个类里，职责不清

**Effort:** 1-2 hours

---

## Approach C: Integrate into TextToImageClient

在现有 `TextToImageClient` 中加 `describe()` 方法。

**Pros:**
- 复用已有 `_build_messages()` 和 `_api_call()`
- 不新建文件

**Cons:**
- 违反单一职责：TextToImageClient 变成"生成+描述"万能类
- Kimi 后端（subprocess）逻辑与 SDK 调用逻辑混在一起
- 现有 client.py 已有 82 行，加描述逻辑后臃肿
- 以后加新后端更难

**Effort:** 1-2 hours

---

## Recommendation

**Approach A** — 用户已在需求阶段确认此架构。协议层隔离后端差异，每个后端独立可测，未来扩展成本最低。`describer.py` 作为新文件，不改动 `client.py`，风险最小。
